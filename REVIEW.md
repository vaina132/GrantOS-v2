# GrantLume (GrantOS-v2) Code Review

## Executive summary

Overall health: solid product surface and good intentions (typed DB client, structured logger, RLS on every table, tuned React Query, hardened cron and DocuSign secrets) undermined by a few serious holes. Two CRITICAL issues stand out: an unauthenticated-by-role member-invite endpoint that allows full multi-tenant takeover, and a Stripe webhook whose signature is verified against a re-stringified body so it is effectively broken or spoofable. Beyond security, the app ships as one un-split bundle, swallows query errors so outages look like empty data and carries heavy schema and type-safety drift. None of this is unrecoverable and most fixes are small to medium. Treat the CRITICAL and HIGH security items as release blockers.

## Findings

1. **[CRITICAL] invite-member has no authorization check (org takeover).** `api/members.ts:77-175` (dispatch `:58-59`, auth gate `:46-55`). The handler authenticates that the caller is some logged-in user but never uses that context: it reads `orgId` and `role` from `req.body` and inserts an `org_members` row with the service-role client, bypassing RLS. Any authenticated user can POST `{email, orgId: <victim>, role: "Admin"}` and become Admin of any organisation, a complete multi-tenant breach. Fix: pass the auth context into the handler and require `auth.orgId === body.orgId && auth.role === 'Admin'` before any mutation. Effort: S.

2. **[CRITICAL] Stripe webhook signature verified against a re-stringified body.** `api/stripe.ts:178-179`, no `bodyParser:false` config in the file. Vercel parses the JSON body, so `JSON.stringify(req.body)` re-serialises it and the bytes no longer match what Stripe signed, so `constructEvent` cannot validate the canonical payload. A forged `checkout.session.completed` or `customer.subscription.updated` with a chosen `org_id` grants free Pro (see finding 6), and legitimate events likely fail too. Fix: add `export const config = { api: { bodyParser: false } }`, read the raw stream into a Buffer and pass that Buffer to `constructEvent`. Effort: S.

3. **[HIGH] DocuSign webhook HMAC verified against a re-stringified body.** `api/docusign.ts:426-427`, no `bodyParser:false` config. Same class of bug as finding 2: the HMAC is computed over a re-serialised object rather than the raw signed bytes, so the otherwise-correct timing-safe check (`:425-434`) cannot match the canonical payload. Either the webhook is non-functional or, if forced to work, timesheets can be marked Signed via forged status callbacks. Fix: disable the body parser and compute the HMAC over the raw request Buffer. Effort: S.

4. **[HIGH] resolve-emails enables cross-tenant PII enumeration.** `api/members.ts:181-207`. Requires only a valid JWT then returns the email for any user UUID supplied in the body, with no org scoping. A user in org A can resolve UUIDs harvested elsewhere to real addresses in org B, a GDPR-relevant leak. Fix: use the auth context and only return emails for user IDs that are members of the caller's org or partners on the caller's projects. Effort: S.

5. **[HIGH] auth-hook authentication is silently optional.** `api/auth-hook.ts:34-40` (CORS `*` at `:27`). The hook secret is only enforced `if (hookSecret)`, so if `AUTH_HOOK_SECRET` is unset the endpoint accepts any unauthenticated POST and sends GrantLume-branded reset and magic-link emails to arbitrary addresses with attacker-chosen token params. Other webhooks here fail closed on a missing secret; this one does not. Fix: treat a missing secret as a hard 500 and restrict CORS. Effort: S.

6. **[HIGH] getPlan always returns 'pro'.** `api/stripe.ts:18-23`. Both branches return `'pro'`, so any price ID (including unknown or foreign ones) and any forged event resolves to full Pro entitlement, decoupling access from the purchased price. Fix: return `'pro'` only when the price ID matches a configured `STRIPE_PRICE_*`, otherwise reject and log. Effort: S.

7. **[HIGH] auth_org_id() and auth_role() use LIMIT 1 for multi-org users.** `supabase/rls_policies.sql:28-34` (also in `schema.sql`), mirrored by `.maybeSingle()` in `api/lib/auth.ts:75-79`. A user belonging to more than one org gets one arbitrary org with no ORDER BY, and every RLS policy keys off that single value, making authorization nondeterministic for such users. Fix: make org context explicit per request (a validated selected-org claim or header) instead of an implicit LIMIT 1. Effort: M.

8. **[HIGH] project_partners pending-invite policy leaks partner PII cross-tenant.** Policy created at `supabase/merge_projects_modules.sql:770-772` (`USING (invite_status = 'pending')` with no org scope), dropped only if `supabase/proposals_workflow_redesign.sql:626` is applied afterwards. As created, any authenticated user can read every org's pending partner rows including contact email and invite token. Survival depends entirely on manual migration order since there is no ordered runner. Fix: add a guarded `DROP POLICY IF EXISTS` to the security catch-all migration and verify it ran on production. Effort: S to verify, M to re-policy.

9. **[HIGH] project_reports partner UPDATE policy has no WITH CHECK.** `supabase/merge_projects_modules.sql:853-858`. An accepted external partner can UPDATE their report rows with no post-image check, so they can rewrite `partner_id` or `period_id` to re-point a report at another partner or period. A broader class of `FOR ALL ... USING` manage policies also lack `WITH CHECK` (`proposal_phases`, `proposal_tasks`, `role_permissions`, `travels` and several `collab_*`), allowing org_id rewrite on update. Fix: add `WITH CHECK` clauses mirroring each `USING` clause. Effort: S for project_reports, M for the sweep.

10. **[HIGH] No route-level code splitting and no manualChunks.** `src/App.tsx:6-35` statically imports all ~30 route pages, and `vite.config.ts:5-18` has no `build.rollupOptions.output.manualChunks`. A first-time visitor to `/login` downloads dashboard, reports, financials, recharts, jspdf and xlsx before they can type a password, producing a multi-MB initial bundle. Fix: convert route elements to `React.lazy` with a Suspense fallback and add a `manualChunks` vendor split. Effort: M.

11. **[HIGH] xlsx and recharts are eagerly bundled into the entry chunk.** `src/lib/exportUtils.ts:1`, `StaffList.tsx`, `ProjectList.tsx`, `ProposalsPage.tsx` and `reportService.ts:2` static-import `xlsx` (~400KB); `Dashboard.tsx:34`, `SalaryCoverageChart.tsx:22`, `ReportWidget.tsx:17` and `BudgetConsumptionChart.tsx:17` static-import recharts (~300KB). Both are parse and eval cost on first paint even when no export or chart is opened. Fix: switch to `await import('xlsx')` inside export handlers (the pattern `reportService.ts:222` already uses for jspdf) and lazy-load the chart components. Effort: M.

12. **[HIGH] checkRecipient issues one listUsers call per recipient and is capped at 1000 users.** `api/send-email.ts:117-156`, looped at `:207-210`. For each recipient it calls `auth.admin.listUsers({perPage:1000})` then a linear find, so a batch send is O(N) serial admin fetches, and any recipient whose auth row is past user 1000 is treated as not found so their opt-out is ignored (a compliance problem). `api/members.ts:94-105` and `:193-200` share the cap. Fix: query `user_preferences` by user_id with a single `.in()` call, or fetch and paginate the user list once into a Map. Effort: M.

13. **[HIGH] React Query error states are never handled, so outages look like empty data.** `grep isError src/features/` returns zero matches although the hooks expose it (`ProjectList.tsx:57`, `Dashboard.tsx:55-63`, `BudgetVsActuals.tsx:30-32`). On a failed fetch `data` falls back to `[]` (e.g. `useProjects.ts:15`) and the page renders "No projects found" as if the user genuinely has nothing. Fix: destructure `isError` and `refetch` in each page and render an error state with retry, ideally via a shared `QueryStateBoundary`. Effort: M.

14. **[HIGH] Pervasive `as any` defeats the typed Supabase client.** 222 `as any` casts across `src`, worst in `src/services/collabProjectService.ts` (65, including `const sb = supabase as any` at `:32`), `timesheetService.ts:9` and `reportTemplateService.ts`. The generated `Database` type and `createClient<Database>` (`supabase.ts:74`) buy nothing in these files, so column renames compile clean and break at runtime. Fix: regenerate `database.types.ts` via the Supabase CLI so the missing tables exist and delete the casts behind the existing row mappers. Effort: L.

15. **[HIGH] Errors swallowed across the codebase while the logger sits unused.** 122 empty or comment-only catch blocks and 137 `} catch {` sites, for example `orgStore.ts:32`, `staffService.ts:23` and `authStore.ts:103,351,384`. `src/lib/logger.ts` is a clean Sentry-integrated logger but only 5 files use it and zero services do, so failures vanish with no log, toast or Sentry event. Fix: route service catch blocks and fire-and-forget `.catch()` handlers through `logger.warn`/`logger.error`. Effort: M.

16. **[HIGH] Timesheet schema drift and an orphaned state-machine trigger.** `supabase/security_audit_fixes.sql:115-159` guards the status-transition trigger on a `timesheet_envelopes` table that no migration ever creates, so the Draft to Approved state machine is never enforced at the DB and any status can jump to any other. The base `timesheet_entries` (`schema.sql:164-183`) also accreted four overlapping models across `migrate_timesheets.sql`, `migrate_timesheet_daily.sql` and `fix_timesheet_envelope_model.sql` with `hours`, `planned_percentage`, `planned_hours`, `actual_hours` and `total_hours` all retained and the status CHECK redefined three times. Fix: attach the trigger to the real envelope table and write one consolidating migration that drops the dead columns and pins a single status CHECK. Effort: L.

17. **[HIGH] AllocationGrid has an N+1 work-package fetch and no cell memoization or virtualization.** `src/features/allocations/AllocationGrid.tsx:181-190` does `Promise.all(projects.map(p => listWorkPackages(p.id)))` (one round trip per project, bypassing React Query), and the cell render at `:913-1001` builds every input inline with fresh closures and no `React.memo` row or windowing. Typing in any cell re-renders the whole grid of thousands of inputs. Fix: batch the WP query with `.in('project_id', ids)` inside a `useQuery` and extract a memoized `AllocationCell` with virtualization via the already-present `@tanstack` libs. Effort: S for the query, L for the grid.

18. **[HIGH] TimesheetGrid bypasses React Query and refetches everything on tab focus.** `src/features/timesheets/TimesheetGrid.tsx:160-188`. Data loads through an imperative `loadData()` in `useEffect` with a `visibilitychange` listener that re-runs the full load on every tab focus, recreating the thundering-herd behaviour that `queryClient.ts` deliberately disabled. The 1472-line component also has almost no i18n. Fix: move loading into `useQuery` hooks and drop the manual visibility refetch. Effort: M.

19. **[HIGH] Core features TimesheetGrid and ProposalPartAForm are entirely un-internationalised.** `src/features/timesheets/TimesheetGrid.tsx` (only 6 `t(` calls, mixed English and German such as the placeholder at `:1284`) and `src/features/proposals/ProposalPartAForm.tsx` (0 `t(` calls, hardcoded option labels at `:59-70`). These daily-use compliance-relevant surfaces ignore the active locale for French, Spanish and Portuguese users. Fix: wire `useTranslation` and move literals to locale keys. Effort: L.

20. **[HIGH] German locale is missing 480 keys and the other locales miss 274 each.** `src/locales/de.json` (1285 keys) vs `en.json` (1743), confirmed by `scripts/check-i18n.cjs --json`. German falls back to raw English for about 28 percent of strings while the strict checker exists (`package.json:14`) but is not enforced in CI. Fix: backfill missing keys with the existing generate scripts and gate regressions with `npm run check:i18n:strict` in CI. Effort: M.

21. **[HIGH] StaffForm validation errors are silently swallowed.** `src/features/staff/StaffForm.tsx`. The zod schema (`:32-51`) validates email, salary, vacation days and an end-after-start refine, but only `errors.full_name` (`:385`) and `errors.fte` (`:464`) are rendered. An invalid email or a backwards date range blocks submit with no message, so the form looks broken. Fix: render a destructive error paragraph for every validated field or add a form-level error summary. Effort: S.

22. **[MEDIUM] CSP allows unsafe-inline and unsafe-eval in script-src.** `vercel.json:17`. Permitting inline-script execution and eval negates most of CSP's XSS value, which matters given the `dangerouslySetInnerHTML` sinks in finding 23. Fix: drop `unsafe-eval` and replace `unsafe-inline` with Vite-injected nonces. Effort: M.

23. **[MEDIUM] Unsanitised dangerouslySetInnerHTML sinks.** `src/features/help/HelpPage.tsx:664,720-729` and `src/features/settings/RolePermissions.tsx:376`. The inputs are static or i18n content today so there is no live XSS, but the sinks are unguarded and the CSP offers no backstop, so any future DB-sourced content becomes stored XSS. Fix: sanitise via DOMPurify or render markdown with a safe renderer. Effort: S.

24. **[MEDIUM] In-memory rate limiting is per-instance and weak at scale.** `api/lib/rateLimit.ts:17-25,46-99`. Counters live in a per-instance Map that resets on cold start and is not shared across the many serverless instances Vercel runs, so the invite-token brute-force and AI/email abuse limits are far weaker than the comments imply. Fix: back security-relevant limits with a shared store such as Upstash Redis. Effort: M.

25. **[MEDIUM] email-preferences reflects any origin on preview deployments.** `api/email-preferences.ts:41-42`. Unlike `api/lib/auth.ts:17-23` which restricts preview origins to a `*.vercel.app` regex, this file reflects any origin when `VERCEL_ENV === 'preview'` with credentials allowed. Fix: reuse the same `*.vercel.app` regex restriction. Effort: S.

26. **[MEDIUM] New project and proposal satellite tables have no org_id column.** `supabase/merge_projects_modules.sql:77-244` and `proposals_workflow_redesign.sql:92+`. Tenant isolation relies on multi-hop RLS subqueries joining back to `projects.org_id` (for example `:778-782`), which run per row on hot tables like `project_report_lines` and offer no independent guard against a mis-set foreign key crossing org boundaries. Fix: add a denormalized NOT NULL `org_id` with an index and simplify the RLS to a direct `org_id = auth_org_id()` check. Effort: L.

27. **[MEDIUM] Foreign keys without covering indexes on several new tables.** `add_deliverables_milestones_reporting.sql` adds deliverables, milestones and reporting_periods with zero indexes on `project_id`, `org_id` or the FK columns; `merge_projects_modules.sql:60`, `add_proposal_phases.sql:22` and `proposals.converted_project_id` similarly lack indexes. The base schema only indexed the original tables. Fix: add the missing `idx_*_project` and `idx_*_org` indexes plus the listed FK indexes. Effort: S.

28. **[MEDIUM] Client-side aggregation that belongs in a DB view or RPC.** `src/services/financialService.ts:152-219` fetches all assignment rows for an org-year with joined salary then sums in JS, with the same pattern for expense actuals (`:225-269`) and `timesheetService.aggregateHoursByYear` (`:745-773`). This transfers whole tables, scales poorly and pulls `annual_salary` to clients who may lack `canSeeSalary` by reading `persons` directly rather than the masked view. Fix: replace with a SQL view or SECURITY DEFINER RPC returning pre-aggregated totals. Effort: M.

29. **[MEDIUM] Money columns use unscaled NUMERIC on the core tables.** `schema.sql` lines 20, 70-71, 99-106, 138 and 213-214 use bare `NUMERIC` for budgets, salary, rates and PMs while newer tables correctly use `NUMERIC(14,2)`, an inconsistency that invites rounding mismatches and is the reason for the JS rounding workarounds in financialService. Fix: alter the money columns to `NUMERIC(14,2)`. Effort: M.

30. **[MEDIUM] persons_masked view may not enforce row-level security.** `schema.sql:79-84`, redefined at `security_audit_fixes.sql:320-330`. The salary-masking view is a plain view, so on Postgres without `security_invoker=on` it runs with the owner's privileges and does not inherit the caller's RLS on `persons`, which could leak other orgs' staff despite the comment asserting otherwise. Fix: set `ALTER VIEW persons_masked SET (security_invoker = on)` and confirm the deployed Postgres version. Effort: S.

31. **[MEDIUM] audit_log entity lookups are unindexed and use TEXT ids.** `schema.sql:247-275`. `entity_id` and `user_id` are stored as TEXT with no FK or index although they reference UUIDs elsewhere, `user_email` is denormalized and goes stale on email change and there is no index on `(entity_type, entity_id)` despite "history for this record" being the natural query. Fix: add `idx_audit_entity (org_id, entity_type, entity_id)` and treat `user_email` as a documented point-in-time snapshot. Effort: S.

32. **[MEDIUM] migrate_timesheet_daily backfill can silently drop hours.** `migrate_timesheet_daily.sql:79-90`. It inserts one placeholder day per old monthly entry dated the first of the month with `ON CONFLICT DO NOTHING`, so two monthly entries for the same person and month collide on the daily unique index and one is dropped without warning. Fix: aggregate old totals per person, project and month before insert and log conflicts instead of ignoring them. Effort: M.

33. **[MEDIUM] CRUD service boilerplate duplicated about 35 times.** Every service in `src/services` hand-rolls the same list, getById, create, update and remove shape with copy-pasted audit strings (for example `projectsService.ts:61,113,125` and `staffService.ts:84,97,109`). Fix: introduce a generic `createCrudService<T>(table, options)` factory. Effort: L.

34. **[MEDIUM] React Query hooks duplicated about 15 times.** `src/hooks/useProjects.ts`, `useStaff.ts` and peers are structurally identical (a list query gated on orgId, a byId query and an invalidate helper). Fix: add a `createEntityHooks(key, service)` generator or a shared `useOrgQuery` wrapper. Effort: M.

35. **[MEDIUM] 16 feature components query Supabase directly, bypassing the service layer.** For example `OnboardingWizard`, `ProfileSettingsPage`, `StaffDetail`, `RolePermissions` and `BulkImport` import `@/lib/supabase` directly, scattering query logic and skipping the audit and masked-persons helpers that the services centralise. Fix: move these queries into the relevant service. Effort: M.

36. **[MEDIUM] ErrorBoundary wraps only authenticated routes and leaks raw error text.** `src/components/common/ErrorBoundary.tsx:50` renders `error.message` directly to users, and the single boundary mounts inside AppShell (`App.tsx:126`) so public pages white-screen on a render error and "Try Again" (`:35`) re-renders the same throwing route. Fix: gate `error.message` behind `import.meta.env.DEV`, wrap the top-level Routes and reset via `resetKeys` tied to `location.pathname`. Effort: S.

37. **[MEDIUM] jspdf statically imported in several modules.** `src/lib/ocdTemplate.ts:1`, `reportGenerator.ts:1` and `timesheetPdfExport.ts:1` import jspdf (~300KB) statically; `reportService.ts:222` already shows the correct `await import('jspdf')` pattern. Fix: dynamic-import jspdf at the export call sites in all three. Effort: S.

38. **[MEDIUM] Five font families declared with full variable axes and one unused.** `src/index.css:3-8` imports two full variable weight axes plus DM Serif and JetBrains Mono, and `@fontsource-variable/inter-tight` (`package.json`) is never imported. These are render-blocking and larger than needed. Fix: subset to the used weights, add `font-display: swap` and drop the unused dependency. Effort: S.

39. **[MEDIUM] exportService fetches tables sequentially and financialService updates row by row.** `src/services/exportService.ts:59-65` and `:92-94` await each table fetch in turn (8 to 13 serial round trips per export), and `financialService.ts:136-143` and `:182-192` issue one awaited UPDATE per changed row. Fix: parallelise the reads with `Promise.all` and batch the writes with a single `upsert`. Effort: S.

40. **[MEDIUM] reportGenerator does heavy synchronous PDF building on the main thread.** `src/services/reportGenerator.ts` has 50-plus nested loops (for example `:751-1078`) building multi-page jsPDF documents synchronously, freezing the UI during export for large projects; `timesheetPdfExport.ts:98-438` is similar. It is correctly dynamic-imported. Fix: chunk the work with `await` yields or move generation to a Web Worker for large datasets. Effort: M to L.

41. **[MEDIUM] 38 leftover console.log statements log PII in production.** `authStore.ts:267` logs every signup response, `ImportDialog.tsx:321-520` has 11 logs including request bodies and sample rows and `AuthCallbackPage.tsx:48-192` has 15 logs printing user emails and invite context. Fix: remove these or route them through an env-gated `logger.info`. Effort: S.

42. **[MEDIUM] absences table carries three sets of overlapping columns.** `schema.sql:188-202` defines `start_date`/`end_date` and `date` and `period`, plus both `notes` and `note`, with nothing enforcing which mode a row uses. Aggregations can silently miss rows depending on which columns a writer populated. Fix: pick one model, migrate data and drop the unused columns including collapsing `note` into `notes`. Effort: M.

43. **[MEDIUM] Hardcoded English in page headers and validation messages.** `TimesheetsPage.tsx:42-43`, `ReportsPage.tsx:45-46`, `Dashboard.tsx:346` and `EnterActuals.tsx:73` use literal titles and toasts, and the zod messages in `StaffForm.tsx:33-51` are English even for the two errors that display. Fix: replace literals with `t()` keys and build the zod schema inside the component so messages can use `t()`. Effort: S.

44. **[MEDIUM] Icon-only timesheet month controls lack accessible labels.** `src/features/timesheets/TimesheetGrid.tsx:759` and `:778` render ChevronLeft and ChevronRight icon buttons with no `aria-label`, so screen-reader users hear an unnamed button on a daily control (part of a broader pattern of about 94 unlabeled icon buttons). Fix: add `aria-label` keys to these two and audit the rest. Effort: S for these, M for the sweep.

45. **[LOW] authStore auto-links persons rows by email match.** `src/stores/authStore.ts:344-352` and the pending-invite `ilike` path at `:441-477` update `persons.user_id` purely by email with no org constraint in the statement. RLS currently bounds this to the caller's org so impact is low, but it trusts email as an identity key. Fix: constrain the link to the resolved org and prefer the server-issued invite token over a raw email match. Effort: S.

46. **[LOW] collab-send computes a wrong invite base URL.** `api/members.ts:333-335`. Operator precedence makes `VITE_APP_URL || VERCEL_URL ? ... : ...` always use `VERCEL_URL`, producing `https://undefined/...` when only `VITE_APP_URL` is set. Fix: parenthesise so `VITE_APP_URL` wins when present. Effort: S.

47. **[LOW] assignments NULL work_package_id defeats the unique constraint.** `schema.sql:142`. Postgres treats NULLs as distinct in a multi-column UNIQUE, so duplicate WP-less allocation rows are not prevented and `allocationsService.ts:71-117` compensates with a racy select-then-insert. `timesheet_days` solved this with a partial unique index. Fix: add partial unique indexes for the `work_package_id IS NULL` case on assignments and pm_budgets. Effort: S.

48. **[LOW] auth.users foreign keys have inconsistent ON DELETE behaviour.** `schema.sql:177-179,250` reference `auth.users` with no ON DELETE clause (defaulting to RESTRICT) while newer actor columns use ON DELETE SET NULL, so user deletion or GDPR erasure will be blocked across many tables. Fix: standardise actor and audit FKs to ON DELETE SET NULL. Effort: S.

49. **[LOW] Migration files embed diagnostic SELECTs and dashboard URLs.** `enhance_absences.sql:63-76` ends with `information_schema` SELECTs and several migrations embed hardcoded project URLs as comments, which is noise for any automated runner. Fix: strip verification SELECTs and hardcoded URLs from migration bodies. Effort: S.

50. **[LOW] Dashboard pinned-report load failures are swallowed.** `src/features/dashboard/Dashboard.tsx:75-79,86-89` use `catch {}` to discard errors, so pinned widgets silently vanish on failure with no way to tell empty from broken. Fix: log the catch to Sentry and optionally show a small load-failed note. Effort: S.

51. **[LOW] Two feature pages are unused dead code.** `src/features/matrix/MatrixPage.tsx` and `src/features/import/ImportPage.tsx` both export a page component with zero import references and no route in `App.tsx` (unlike `timeline/TimelinePage.tsx`, which is intentionally kept for the `/timeline` redirect). Fix: remove both files or wire them into routing if still intended. Effort: S.

52. **[LOW] Ten exported helpers in utils.ts are never imported.** `src/lib/utils.ts` exports `formatCurrencyDetailed` (`:20`), `formatDateShort` (`:39`), `formatNumber` (`:48`), `formatPms` (`:53`), `formatPercentage` (`:58`), `getMonthName` (`:105`), `getMonthsInYear` (`:110`), `clampPms` (`:122`), `getSalaryRecoveryColor` (`:126`) and `getSalaryRecoveryBgColor` (`:132`), each with zero grep hits elsewhere in `src`. Fix: delete the unused functions (the live duplicate date and number helpers from finding 33 can replace any future need). Effort: S.

## Quick wins

The five best value-to-effort items:

1. Finding 1: add the Admin and org check to invite-member. One small guard closes a full org-takeover hole. Effort S.
2. Finding 2 with 3: disable the body parser and verify Stripe and DocuSign webhooks over the raw Buffer. Fixes spoofable billing and signing in two near-identical edits. Effort S.
3. Finding 4 with 5: scope resolve-emails to the caller's org and make auth-hook fail closed on a missing secret. Two small changes that stop PII enumeration and an open branded mailer. Effort S.
4. Finding 13: surface `isError` and a retry across the data pages (ideally one shared wrapper). High user-trust payoff for medium effort. Effort M.
5. Finding 10: route-level `React.lazy` plus a `manualChunks` vendor split. The single biggest load-time win for medium effort. Effort M.
